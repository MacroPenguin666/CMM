# CMM Repo Restructuring Plan

## Context

The CMM repo has grown organically: 5 separate SQLite databases (137MB total, 71 tables), code scattered across `scripts/policy_monitor/`, `scripts/customs_scraper/`, `live/`, and 8 top-level `run_*.py` scripts. The goal is:

- **Backend/frontend separation** вЂ” one folder for data acquisition, one for the dashboard
- **Single consolidated database** вЂ” merge 5 SQLite DBs into one
- **Private external hosting** вЂ” Supabase (PostgreSQL) as the central DB, fully private (no public API keys)
- **Static GitHub Pages dashboard** вЂ” a GitHub Action exports display-ready JSON from Supabase; no credentials in the frontend
- **Incremental fetching** вЂ” only fetch/write missing data (already implemented in most fetchers)

**Data flow:**
```
Local fetchers в†’ Supabase (private, service key)
                      в†“
              GitHub Action (cron)
              queries Supabase в†’ exports JSON
                      в†“
              frontend/data/*.json (committed)
                      в†“
              GitHub Pages serves static dashboard
```

## Phase 3: Supabase Setup (Private)

Set up Supabase as the authoritative external database. Fully private вЂ” no public API access.

### Work items

1. **Create Supabase project** (free tier: 500MB, unlimited API requests)

2. **Convert schema from SQLite to PostgreSQL:**
   - `INTEGER PRIMARY KEY AUTOINCREMENT` в†’ `SERIAL PRIMARY KEY`
   - `INSERT OR IGNORE` в†’ `INSERT ... ON CONFLICT DO NOTHING`
   - `datetime('now')` в†’ `NOW()`
   - `REAL` в†’ `DOUBLE PRECISION`
   - ~15 instances to change (grep for SQLite-isms)

3. **Run DDL in Supabase SQL editor** to create all ~63 tables

4. **Disable all public access:**
   - Enable RLS on every table: `ALTER TABLE <t> ENABLE ROW LEVEL SECURITY;`
   - Create NO policies for the anon role (default-deny)
   - Only the service-role key can read/write

5. **Add Supabase credentials to `.env`:**
   ```
   SUPABASE_URL=https://xxxxx.supabase.co
   SUPABASE_SERVICE_KEY=eyJ...  # service role вЂ” never exposed publicly
   ```

6. **Add dual-write to `backend/storage.py`:**
   - Keep local SQLite writes (backup + offline dev)
   - Add Supabase writes via `supabase-py` SDK
   - `upsert_df()` writes to both local SQLite and Supabase
   - `get_latest_date()` queries Supabase (source of truth) with SQLite fallback

7. **Write `backend/sync_to_supabase.py`** вЂ” one-time bulk upload of existing local data to Supabase

8. **Add `supabase` to `requirements.txt`**

**Verify:** Run `python backend/sync_to_supabase.py`; check Supabase dashboard shows all tables with correct row counts; run a fetcher and confirm new rows appear in both local SQLite and Supabase.

### GitHub Action (`.github/workflows/export-data.yml`)

```yaml
name: Export data & deploy dashboard
on:
  schedule:
    - cron: '0 */4 * * *'    # every 4 hours (matches news fetch)
  workflow_dispatch:           # manual trigger
  push:
    branches: [main]
    paths: ['frontend/**']     # redeploy on dashboard changes

jobs:
  export:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install supabase
      - run: python export.py
        env:
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY }}
      - uses: actions/upload-pages-artifact@v3
        with: { path: frontend }

  deploy:
    needs: export
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
    steps:
      - uses: actions/deploy-pages@v4
```

### Add GitHub Secrets
- `SUPABASE_URL` вЂ” the project URL
- `SUPABASE_SERVICE_KEY` вЂ” the service role key (never exposed in code)

## Phase 5: Frontend Rewiring

Modify `frontend/index.html` to load data from JSON files instead of Flask API endpoints.

### Work items

1. **Replace the `api()` helper:**
   ```javascript
   // OLD:
   const api = p => fetch(p).then(r => r.json());

   // NEW:
   const api = p => {
       const map = {
           '/api/news':                    './data/news.json',
           '/api/news/sources':            './data/news_sources.json',
           '/api/news/stats':              './data/news_stats.json',
           '/api/financial/snapshots':     './data/financial_snapshots.json',
           '/api/financial/indicators':    './data/financial_indicators.json',
           // ... one entry per endpoint
           '/api/overview':                './data/overview.json',
       };
       const file = Object.entries(map).find(([k]) => p.startsWith(k));
       return fetch(file ? file[1] : p).then(r => r.json());
   };
   ```

2. **Handle parameterized endpoints:**
   Some endpoints take query params (e.g., `/api/financial/series/SHIBOR_ON`, `/api/trade/map?year=2023&flow=exports`). The export script pre-computes these:
   - Financial series: one JSON file per indicator, or a single `financial_series.json` keyed by indicator
   - Trade map: pre-computed for each (year, flow) combination, or a single large file the JS filters client-side
   - Macro series: one large `macro_series.json` grouped by variable

3. **Update GeoJSON paths:**
   ```javascript
   // OLD: fetch('/api/geo/provinces')
   // NEW: fetch('./data/geo/china_provinces.json')
   ```

4. **Remove the academic voting feature** (POST endpoint вЂ” can't work on a static site)

5. **Remove/static-ify flight and ship live polling** вЂ” show last-known positions from the exported JSON, no periodic refresh

6. **Remove the AI advisor endpoint** (POST to Anthropic API вЂ” needs a server)

**Verify:** Open `frontend/index.html` directly in the browser (file:// or a local HTTP server). All 8 tabs load and display data correctly. No network requests to localhost:5001.

## Security Summary

| Layer | Access | Protected by |
|-------|--------|-------------|
| Supabase DB | Service key only | RLS enabled, no anon policies, key in `.env` + GitHub Secrets |
| Export script | Runs in GitHub Actions | Service key is a GitHub Secret, never in code |
| Frontend JSON | Public (GitHub Pages) | Contains only display-ready data вЂ” same as what's visible in the dashboard |
| Frontend HTML | Public (GitHub Pages) | No credentials, no API keys, no database access |

No one can query the database. The only data exposed is what the dashboard already shows.
